从 Agent Trajectory 到 Agent Skill：如何把历史经验变成复利

如果把 agent memory 拆开看，其实有两条不同的主线。 第一条是 personal memory：agent 如何更懂你，理解用户偏好、长期上下文、隐含约束和历史决策。这类问题更接近 long-context QA，评估的重点往往是“它是否记得、是否理解、是否能在正确场景下调用用户相关信息”。 第二条是 procedural memory：agent 如何避免重复犯错，如何从过去的成功与失败中总结出可复用经验，如何越跑越好、越用越强。它更关心的是 agentic benchmark 上的任务成功率、工具调用效率、错误率下降、 token的成本下降，以及在真实环境中是否能形成工程复利。 这篇文章主要讨论第二条路线。 今天很多 agent-harness， Claude Code、Hermes-agent、OpenClaw等，在长期使用后都会积累大量 trajectory这些轨迹实际上记录了 agent 和环境交互中最宝贵的东西。 但问题是，人类对这些 trajectory 的利用率尚且不足。大多数时候，trajectory 被当成日志、debug 记录，或者最多作为检索材料塞回上下文。一个 agent-harness 跑了半年，可能积累了成千上万条任务轨迹，但下次遇到相似问题时，agent 仍然可能从零开始探索，重复踩同样的坑。 引出了一个核心问题： trajectory 到底如何才能变成 skill？ 以 Hermes-agent 这类系统为例，用久了之后会发现 skills 可能越来越多。但真正有效的 skills 有多少？ 本文将借最近几篇相关工作，讨论一个正在变得重要的问题：如何从大量 agent trajectories 中，自动挖掘、总结、验证并优化可以被未来任务复用的 agent skills。 以hermes-agent 作为入口 Hermes-agent 有一个很有意思的定位：一个会随着使用而变强的 agent。也就是把过去的经验沉淀为可以被未来复用的外部记忆。这里的外部记忆大致可以分成两类： 第一类是 declarative memory，也就是关于用户、项目、环境和偏好的事实记忆，具体形式是 USER.md 和MEMORY.md， 这类 memory 解决的是“agent 如何更懂你”的问题。 第二类是 procedural memory，也就是 skills。具体形式是写入到state.db 中的 sessions/messages，以及被总结成 md 的skills。 ...
