新模型#

Meta Llama3.3#

Meta Llama 3.3 是一款700亿参数的指令微调多语言大模型。该模型专注于文本生成，相比 Llama 3.1 70B 和 Llama 3.2 90B 性能有显著提升。在某些应用场景中，其效果可与 Llama 3.1 405B 媲美，但模型体积更小、计算效率更高。

核心亮点

  1. 高性能表现：支持复杂推理、代码生成和多语言处理。在 HumanEval 等基准测试中表现优异，代码生成 pass@1 达到88.4%。
  2. Grouped-Query Attention (GQA)：能处理128,000 tokens的长上下文，大幅提升推理效率。
  3. 无预训练版本：Llama 3.3 专注于指令微调版本，更好地适配实际应用场景。
  4. 与 Llama 3.1 兼容：完全支持 Llama 3.1 的提示模板、代码解释器和工具调用功能，可实现无缝迁移。

huggingface: https://huggingface.co/meta-llama/Llama-3.3-70B-Instruct/tree/main

Gemini 2.0#

Gemini 2.0 是谷歌推出的下一代多模态人工智能模型。相比 1.5 Pro，Gemini 2.0 Flash 在多模态处理方面有重大突破，大幅提升了文本、代码、视频、空间理解与推理性能，同时维持低延迟和高效率。在关键基准测试中，该模型表现卓越，处理速度达到前代的两倍。

创新亮点

  • 原生多模态输出：通过单一 API 调用，可生成包含文本、音频和图像的综合响应，极大简化了开发流程。
  • 实时多模态 API：支持摄像头和屏幕流输入，集成语音活动检测等自然交互功能，特别适合实时应用开发。
  • 原生工具调用：能够直接调用 Google 搜索、执行代码并对接第三方函数，有效支持自主代理系统的构建。
  • 长上下文与推理：优化了复杂任务处理、长文本理解与规划能力，特别适合研究助手类应用。

实际效果

开发者可借助 Gemini 2.0 构建实时音视频处理的动态应用。模型能够直接处理无字幕视频并生成精确翻译和结构化文章，大大简化了传统的多步骤流程。Project Astra 展示了模型在多语言对话、智能记忆和快速响应方面的优势，而 Project Mariner 在浏览器自动化任务中更是达到了 83.5% 的领先性能。

Qwen QVQ#

QVQ 是一个基于 Qwen2-VL-72B 开发的多模态推理模型，主要面向视觉理解和复杂问题求解。在 MMMU 基准测试中，QVQ 获得了 70.3 分的成绩，相比此前的模型有明显提升，特别是在跨学科视觉理解方面。

模型的核心优势在于其细致的推理能力，尤其体现在数学类任务中。在 MathVista 和 MathVision 等数理推理测试中，QVQ 展现出强大的问题分析能力。同时，在包含高中奥赛级别双语科学题的 OlympiadBench 测试中，模型也取得了不错的表现。

目前，QVQ 仍面临一些技术挑战，包括多语言场景下的准确性问题、推理过程中的循环冗余，以及在复杂多步骤任务中的逻辑连贯性等。这些问题提示了视觉语言模型在实际应用中需要进一步改进的方向。

QVQ: To See the World with Wisdom

DeepSeek-V3#

深度求索公司最新推出的大语言模型 DeepSeek-V3 是当前 AI 领域的一项重要突破。这款模型采用专家混合架构，总参数量达到 6,710 亿，其独特之处在于能为每个处理单元灵活调用 370 亿参数。

在技术创新方面，DeepSeek-V3 采用了无辅助损失的负载均衡方案，有效解决了传统均衡策略带来的性能损耗。团队还创新性地引入多 token 预测机制，不仅提升了模型整体性能，还为加快推理速度开辟了新途径。

在实际应用中，DeepSeek-V3 表现出色。它在数学运算和程序编写等专业领域的性能不仅超越了现有开源方案，还可与顶尖商业模型媲美。特别值得一提的是，它将文本生成速度提升至每秒 60 个 token，让用户能享受近乎实时的流畅对话体验。

从经济效益角度看，DeepSeek-V3 在成本控制方面表现优异。整个训练过程仅消耗 278.8 万小时的 H800 GPU 算力，总投入约 557 万美元，在大模型研发领域可谓高效。

目前，DeepSeek-V3 已在 Hugging Face 平台上线，并向全球开发者开放源代码。模型支持 FP8 原生权重，并提供便捷的格式转换工具，为社区开发者的二次开发和应用创新提供了极大便利。

Introducing DeepSeek-V3

Genie 2#

deepmind 的 Genie 2实现了一项突破性进展：能从单张图像生成完整的3D交互环境。这个创新的世界模型不仅可以理解和重现图像中的场景元素，还能将其转化为沉浸式的人机交互环境。

主要特点：

  • 沉浸式交互：用户通过简单的键鼠操作可与环境实时互动，系统能准确模拟物体的物理特性，包括碰撞和重力等自然现象。
  • 场景连续性：系统能在视野外的区域保持环境的一致性和连贯性，营造真实可信的探索体验。
  • 多样化生成：系统可基于同一起始场景演化出多种不同的发展路径，为AI训练提供丰富的数据样本。

系统采用先进的自回归潜变量扩散模型，结合动态Transformer架构，实现流畅的实时场景生成和精确的交互响应。

Genie 2为AI研究和创意开发开辟了新途径。它简化了虚拟环境的构建流程，同时为智能体训练提供丰富的测试场景，推动具身智能研究的发展。

新技术和更新#

Anthropic MCP#

Model Context Protocol（MCP）是一个创新的开放标准，专门解决AI助手与外部系统交互时的核心问题。简单来说，MCP就像一个通用翻译器，使AI模型能够自然地与各类数据源和工具"对话"。

在MCP出现前，AI助手常遇到"失忆"和"语言不通"等问题：它们难以保持对话连贯性，且需要开发者耗费大量精力来处理不同数据源的兼容性。这就像同时与使用不同方言的人交谈，每次都需要专门的翻译。

MCP的推出从根本上改善了这一状况。它建立了一套通用的"交互规则"，让AI助手能够：

  • 准确理解并记住用户的需求背景
  • 轻松获取各类数据源的信息
  • 灵活调用不同的工具和服务

对开发者而言，MCP大幅简化了开发流程。以往需要为每个外部工具编写专门的接口程序，而现在只要遵循MCP标准就能实现快速对接。这相当于把定制"专线"升级为标准化"高速公路"。

对最终用户来说，这带来了更智能、更流畅的使用体验。AI助手不仅能更好地理解用户意图，还能保持对话连贯性，并在需要时无缝切换各种工具完成复杂任务。

从行业发展角度看，MCP正在打造一个开放共赢的生态系统。它为AI应用开发确立了统一的"游戏规则"，推动技术创新和行业协作，引领AI行业迈向更成熟的发展阶段。

OpenAI 连续12天发布#

2024年12月4日至12月20日，OpenAI连续12天举行产品发布会，推出多项新产品和功能。以下是每日发布内容的整理：

  1. 第1天（12月4日）：发布o1模型完整版，性能较预览版显著提升，错误率降低34%，支持多模态输入。同时推出ChatGPT Pro月付200美元订阅服务，提供无限制访问o1和专业版o1的权限。
  2. 第2天（12月5日）：推出强化微调技术（Reinforcement Fine-Tuning），让用户能用少量数据在特定领域（如法律、金融、医疗等）创建专家模型。
  3. 第3天（12月6日）：推出AI视频生成工具Sora正式版和Sora Turbo，可生成最长20秒、最高1080p的视频，并提供故事板、Remix、Blend和Loop等编辑功能。
  4. 第4天（12月7日）：全面开放ChatGPT Canvas，提供人机协作的写作和编程平台，支持与ChatGPT共同编辑文档和代码，内置Python模拟器实现实时协作。
  5. 第5天（12月8日）：ChatGPT正式集成到Apple Intelligence，用户可通过Siri使用ChatGPT功能，增强写作工具和视觉智能能力。
  6. 第6天（12月9日）：为ChatGPT高级语音模式增加视频输入和实时屏幕共享功能，使其能"看到"用户和周围环境，提供更丰富的交互体验。
  7. 第7天（12月10日）：推出Projects功能，支持创建特定项目、上传文件、设置自定义指令，并集中管理项目相关对话，增强项目管理和协作能力。
  8. 第8天（12月11日）：升级ChatGPT搜索功能，提升搜索速度，支持多模态输出，向全球所有登录用户免费开放，提供实时网络信息获取能力。
  9. 第9天（12月12日）：o1模型正式推出API，成本降低60%，新增函数调用、开发者消息和图像分析等功能，方便开发者集成使用。
  10. 第10天（12月13日）：开通电话和WhatsApp使用ChatGPT的功能，用户可拨打特定号码或通过WhatsApp与ChatGPT进行语音或文字对话。
  11. 第11天（12月14日）：宣布ChatGPT与Mac应用深度集成，特别是在编程和写作方面，可与Warp、Xcode、Notion等应用协同工作，提升生产力。
  12. 第12天（12月15日）：推出新一代推理模型o3及其轻量版o3 Mini，在数学推理、编程效率和科学问题解决等方面展现强大能力，标志着AI推理能力的重大突破。

Building effective agents#

At Anthropic, we categorize all these variations as agentic systems, but draw an important architectural distinction between workflows and agents:Workflows are systems where LLMs and tools are orchestrated through predefined code paths.Agents, on the other hand, are systems where LLMs dynamically direct their own processes and tool usage, maintaining control over how they accomplish tasks.

When more complexity is warranted, workflows offer predictability and consistency for well-defined tasks, whereas agents are the better option when flexibility and model-driven decision-making are needed at scale.

We suggest that developers start by using LLM APIs directly: many patterns can be implemented in a few lines of code. If you do use a framework, ensure you understand the underlying code. Incorrect assumptions about what’s under the hood are a common source of customer error.

To repeat: you should consider adding complexity only when it demonstrably improves outcomes.

Start with simple prompts, optimize them with comprehensive evaluation, and add multi-step agentic systems only when simpler solutions fall short.

Frameworks can help you get started quickly, but don’t hesitate to reduce abstraction layers and build with basic components as you move to production.
