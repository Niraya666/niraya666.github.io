值得关注的研究和论文#

Do Language Models Need Sleep? Offline Recurrence for Improved Online Inference#

arXiv:2605.26099

论文提出一种类似“睡眠”的离线记忆巩固机制：长上下文模型不只是要保留被挤出窗口的内容，更要在内容被清空前做额外计算，将其整理成后续推理可用的内部记忆。

Offline Recursive Memory Consolidation

当上下文窗口满了、KV cache 即将被丢弃前，模型进入 sleep-like consolidation 阶段，对当前 chunk 进行多次 recurrent forward pass，并反复更新 SSM block 里的 fast weights；随后清空 KV cache，继续处理后续 token。

关键点是把额外计算从在线预测时挪到离线 consolidation 时，保持 wake-time prediction 的单步延迟。实验覆盖 cellular automata、多跳图检索和数学推理任务，结果显示 sleep duration 越长，模型在需要深层推理的样本上收益越明显。

Compiling Agentic Workflows into LLM Weights: Near-Frontier Quality at Two Orders of Magnitude Less Cost#

arXiv:2605.22502

论文讨论一个很直接的问题：对于稳定、可流程化、需要长期复用的 agent workflow，是否可以不在运行时显式编排，而是在训练阶段把流程蒸馏 / 编译进小模型，让小模型自己学会 self-orchestrate。

Subterranean Agent

传统 surface orchestration 是“用户 -> orchestrator -> LLM”，每一轮都由外部 orchestrator 注入流程、路由和决策；subterranean agent 则只在训练数据生成阶段使用 orchestrator，部署时用户直接和 fine-tuned LLM 对话。

作者在旅行预订、Zoom 支持、保险理赔等流程上生成 workflow trajectories，再用 SFT 将流程压入小模型。结果显示，在稳定、有限、模拟分布内的客服型流程中，小模型可以在对话质量上接近显式 orchestration，同时显著降低成本。

边界

这不等于小模型真正学会了可验证 workflow engine 的抽象执行能力。它更像是把高频、稳定、结构化的流程压缩进参数中；一旦流程开放、规则变化频繁或分布外输入增多，显式 orchestration 和可审计状态机仍然更可靠。

Adapting the Interface, Not the Model: Runtime Harness Adaptation for Deterministic LLM Agents#

arXiv:2605.22166

论文指出很多 Agent 失败并不是模型“不会想”，而是模型和环境之间的 runtime interface / harness 设计得不好。与其微调模型，不如改造 Agent 和环境之间的接口层。

Life-Harness 冻结 LLM 参数，不修改 benchmark 环境，只从训练轨迹中总结可复用的 harness 干预规则，然后在测试任务上固定使用。

把 Agent Adaptation 从“改模型”转向“改接口”

传统做法会把环境规则、工具习惯、恢复策略都吸收到模型参数里；Life-Harness 则把这些东西显式放在 harness 层。作者称之为 runtime interface adaptation：模型参数固定，环境固定，评测协议固定，只改中间的 runtime harness。

四层 Lifecycle-Aware Harness

  1. Environment Contract Layer：交互前，把稳定环境约束、工具使用规则、业务 policy、常见坑显式写清楚。
  2. Procedural Skill Layer：任务开始时，从训练轨迹中蒸馏 procedural skills，并根据当前任务检索相关 skill。
  3. Action Realization Layer：模型输出 action 后、环境执行前，检查 action 是否可执行，并做 deterministic canonicalization 或 blocking。
  4. Trajectory Regulation Layer：环境反馈后，监控轨迹是否重复、停滞、无效重试、接近预算耗尽，并触发 recovery。

从失败轨迹中演化 Harness，而不是训练模型

作者用 Qwen3-4B-Instruct 在训练任务上跑出失败轨迹，然后用 coding agent 读取轨迹、诊断失败模式，并迭代修改 harness。测试集始终隐藏，最终 harness 在 evaluation 时固定，不再在线学习。

整体流程可以理解成：

收集轨迹 -> 诊断失败类型 -> 归因到生命周期层 -> 修改 harness -> 重新跑训练任务 -> 检查提升和 regression -> 固定 harness -> 测试迁移。

Harness Updating Is Not Harness Benefit: Disentangling Evolution Capabilities in Self-Evolving LLM Agents#

arXiv:2605.30621

自进化有用时，到底是谁的功劳？是 evolver 写得好，还是 task-solving agent 用得好？这篇论文把 self-evolution 拆成两个能力来分析。

两个能力

  • Harness-updating capability：模型作为 evolver 时，能否根据执行证据写出有用的持久化 harness 更新。
  • Harness-benefit capability：模型作为 task-solving agent 时，能否在任务执行中真正利用更新后的 harness 获益。

论文发现两者并不一致：会写更新，不等于能从更新中获益。

实验设置

作者把多个模型分别放在两个角色里：一部分模型作为 task-solving agent，一部分模型作为 evolver；固定任务流、初始 harness、prompt 模板、进化预算和 turn limit，然后观察不同 agent-evolver 组合的效果。

主要发现

  • Harness-updating 能力和基础模型能力关系很弱：不同 evolver 之间差距很小，任一 benchmark 上最好和最差 evolver 的差距最多只有 3.1 个百分点。
  • Harness-benefit 能力和基础模型能力呈非单调关系：弱模型因为不会加载或遵循 harness，收益很低；强模型因为原本已经很强，接近性能天花板，收益也有限；反而是中等能力模型最受益。

Self-Harness: Harnesses That Improve Themselves#

arXiv:2606.09498

Self-Harness 让 LLM Agent 根据自己的失败轨迹，自动改进它运行时依赖的 agent harness。论文强调，同一个 base model 在不同 harness 下表现可能差异很大；模型权重不变，变的是外围执行规则与运行机制。

受控自改进循环

Self-Harness 的循环分为三个阶段：

  1. Weakness Mining：从执行失败中挖出可复用失败模式。
  2. Harness Proposal：提出小范围、可解释的 harness 改动。
  3. Proposal Validation：通过回归测试决定是否接受候选改动。

从失败案例到失败机制

论文把失败分析从“单个失败案例”提升到“聚类后的失败机制”：根据 verifier 暴露的终止失败原因、agent 行为的因果状态、抽象行为机制形成 failure signature，再把相同 signature 的失败聚到一起。

工程启发

Self-Harness 强调 minimal and auditable edits。每个候选改动都要说明来源和作用，最终接受的改动也都是可记录、可审计、可回滚的小改动，而不是一次性重写整个 agent 架构。

Can LLM Agents Infer World Models? Evidence from Agentic Automata Learning#

arXiv:2606.16576

LLM Agent 在与环境交互时，到底能不能逐步推断出隐藏的“世界模型”，还是只是靠局部模式和浅层启发式行动？论文用自动机学习构造了一个可控评测框架。

Agentic Automata Learning

让 Agent 通过工具调用去还原一个隐藏的确定性有限自动机 DFA。DFA 就是实验里的“世界模型”。Agent 可以问两类问题：

  • Membership query：问某个字符串是否被目标 DFA 接受。
  • Equivalence query：提交一个候选 DFA，问它是否等价于隐藏 DFA；如果不等价，oracle 返回一个反例字符串。

评测价值

  • 把经典自动机学习改造成面向 tool-calling LLM agent 的评测框架。
  • 提供相对 TTT 的额外工具调用数、规划失败 vs 推理失败、non-informative query 比例、Best Hypothesis Similarity 等过程指标，比单纯 pass/fail 更能解释 Agent 失败在哪里。
  • 任务可程序生成、可扩展、无需人工标注，适合研究 Agent 是否具备交互式发现和世界模型归纳能力。

OpenClaw-Skill: Collective Skill Tree Search for Agentic Large Language Models#

arXiv:2606.16774

论文认为“技能”可以封装可复用的程序性策略，比如如何调用工具、如何验证中间结果、如何从错误中恢复。当前 skill 方法的核心问题是碎片化、多样性不足和跨模型迁移弱。

Collective Skill Tree Search

核心创新是 Collective Skill Tree Search（CSTS）：让多个异构模型分别尝试解决同一个子任务，然后把它们的执行轨迹总结成候选技能节点，以缓解单模型偏差，并覆盖更多规划、执行、诊断、恢复模式。

技能筛选

候选 skill 一方面由多个 judge model 从清晰性、可执行性、完整性、相关性等维度打分；另一方面做“迁移性评分”，即把某个模型产生的 skill 给其他模型使用，看它是否真的能帮助其他模型完成同一子任务。最终分数是质量分和迁移分之和。

Collective Skill Reinforcement Learning

CSRL 不是只基于单个 skill 做 RL，而是对同一个子任务采样多个 skill-conditioned rollouts，并在整个 skill group 内比较奖励，让模型学会偏好更有效的技能策略，避免被单一 skill 诱导到同质化或次优解。

具体流程

  1. 任务分解：给定复杂任务 T，CSTS 先分解成有序子任务，每个子任务对应技能树的一层。
  2. CSN-Gen 生成技能节点：多个模型分别执行同一子任务，产生包含中间推理状态、动作、观察/执行反馈和最终结果的轨迹；共享 skill synthesizer 将每条轨迹总结成一个技能节点。
  3. CSN-Assess 筛选技能：先由多个 judge model 做质量评分，再让其他模型试用该 skill，计算迁移性评分。
  4. 构造 skill-augmented SFT 数据：选出的组合式 skill path 与对应 demonstration trajectory 组合成监督微调样本。
  5. CSRL 强化学习：在同一子任务的多个 skill-conditioned rollouts 之间做 group-relative advantage，并用 GRPO-style clipped objective 更新策略。

从工程师到 Staff：技术影响力到底是什么#

这个问题会被抽象成职级：Senior、Staff、Principal。技术人的价值不再主要来自自己完成多少任务，而是来自能否影响更大范围的问题；从建立长期技术方向，到让团队整体产出变好。真正的技术影响力不是权力，而是信任。

Staff 级别的人需要更往前一步：判断任务背后的问题是否定义正确。定义问题比解决任务更难，因为它要求你理解上下文。技术影响力还体现在系统权衡上，越复杂的系统，越需要有人把这些 trade-off 摆到台面上。

Staff 的价值不能只靠自己写更多代码。Staff 的产出，很多时候体现在别人能更好地产出。

如果说优秀工程师解决问题，那么 Staff 级技术人要做的是：帮助团队解决正确的问题，并让解决问题的能力持续变强。

Why AI hasn’t replaced software engineers, and won’t#

There is great anxiety and uncertainty about AI replacing jobs.

Why coding agents haven’t led to labor displacement: the decide-execute-deliver sandwich.

(1) deciding and specifying what to build, (2) verifying and being accountable for what is delivered, and (3) the deep human understanding.

In fact, not only are software engineering jobs not going away due to AI, there might even be an increase in demand for software engineers.

观点 | 从 Vector Database 到 Vector Lakebase，如何定义 AI data infra 的下一个十年#

数据库的演化基本都遵从同一个逻辑：先承接流量，再挖掘价值，最后统一架构消除割裂。

非结构化数据真正需要的不只是检索系统。它是一个需要持续闭环的行动系统：CS/CD，Continuous Serving + Continuous Discovery。

这两大服务不是独立工作流，而是一个飞轮。Serving 产生反馈，Discovery 改进质量，改进后的数据和索引再回到 Serving，循环往复。

CS/CD 飞轮要转得起来，第一个硬约束是：不同计算模式不能各自维护一份业务数据副本。

Vector Lakebase 建设的关键包括存算分离 + 多层缓存、统一的多模态非结构化数据管理、原生向量数据库能力、多种计算模式、开放格式和数据不锁定。

组织架构的本质：信息处理的分工与协调#

科层制存在，是因为人的信息处理能力有限。当 AI 打破了这个限制，科层制存在的理由就不成立了。科层制的本质，就是一个信息处理的分工与汇总系统。

科层制能运转，依赖三个隐含假设：执行者不需要理解全局；信息向上必须汇总和压缩；决策向下必须分解和指令化。这三个假设的共同基础是：人的信息处理能力有限。

让信息处理能力的供给从“人的带宽”变成“算力的规模”之后，执行者可以理解全局，信息不再需要逐级压缩，决策也不再需要逐级分解。

网状组织要运转，有一个不可忽视的前提：每个人都需要有足够的判断力和自主性。组织形态的转变，和人的能力模型的转变，是耦合的。

为什么东亚人如此害怕裸辞？#

对于很多人来说，“临时”是一种很可怕的状态。

在文学里，恰恰是那些偶然的、临时性的东西，会绽放出更多的生机。相反，永恒、绵延、不朽，反而变成了令人恐惧的东西。

一个人一旦被许诺“永远存在”，反而会消解他在有限世界里所能产生意义的一切行为和思考。

OpenAI、Anthropic 都开始押注 FDE，FDE 才是 Agent 时代的 PMF 范式？#

为什么 agent 时代所有人都在抢 FDE？因为客户买的不是软件，而是一个能干活的 agent。

FDE 带着现有的产品进场，去填补产品能力与实际需求之间的 gap，把路线先铺成一条“碎石路”。总部的产品与工程团队再把这些现场做法抽象、泛化，修成能服务接下来五到十个客户的“高速公路”。

FDE 团队在每个客户现场所做的事情确实有点像是一个创始人，但 FDE 成员的区别是他们可以接触到一些非常强大的 product leverage。

产品团队的角色是去真正把握 product vision。看到一个新问题时，他们必须思考它的通用形态会是什么样，什么版本可以应用于接下来的 10 个客户。

YC 有一个概念叫 Collison install：不要等着人们来你的网站，要去找他们，让他们安装软件，亲自去他们那里，坐在他们旁边。

GPT-4 给他带来的是一次宗教体验。他说当时的感受是：如果我们不做 AI，那今天做的一切都会失去意义。于是 Notion 再次开始重建。

AI 产品开发是技术优先，而不是客户优先。你无法精确控制最终结果，只能不断实验、观察模型表现、调整方向。

未来的人才 = Capability × Taste × Agency。

不要试图在所有地方创新，因为组织资源有限。如果产品创新、销售创新、市场创新以及管理创新同时进行，大概率会全部失败。

致超级个体 | To The Crazy Ones#

超级个体不是“更强的岗位人”，而是能够重新拿回完整 closed-loop 的人。

当你们讨论“AI 时代的人才战略”时，真正要问的不是“谁会用 AI”，而是“组织有没有允许一个人从问题发现，一直走到拿到结果”。

人的能力明明已经被工具放大了，组织却仍然把他切成一个局部函数。

LLM 和 Agent 不是简单提高某个岗位的效率，而是把被拆开的能力重新推回一个人身边。

AI 工业革命早期最有价值的探索，恰恰常常没有稳定名字。它一开始不是“平台”，不是“战略”，不是“创新专项”，只是一个员工觉得“这事应该可以自动化”的冲动。

超级个体首先不是一组技能包，而是一种底层人格结构：好奇心强，好胜心也强；创新精神强，同时也是别人眼中的 trouble makers；自学能力强；自驱力强；动手能力强。

真正关键的不是 Agent 变多，而是人重新站回最终责任人的位置。真正的挑战只有一个：能不能把这些资源重新接到更小的闭环上。
